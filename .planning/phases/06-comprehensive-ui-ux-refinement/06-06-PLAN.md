=== ISSUE 1: CALENDAR APPEARING CUT OFF / CLIPPED ===

Root cause investigation reveals:
- Global CSS (/app/globals.css): NO overflow:hidden anywhere
- Admin layout (/app/admin/layout.tsx): NO overflow:hidden, no height constraints
- Dashboard page (/app/admin/page.tsx): Uses <section className="grid gap-6 lg:grid-cols-2"> — standard responsive grid. NO overflow:hidden.
- NO height constraints, NO overflow:hidden in ANY parent container from <html> down to DashboardCalendar section

LIKELY ROOT CAUSE: The expanded day popup is positioned with absolute z-index but has NO overflow constraint. The popup uses:
  className="absolute top-full left-0 z-20 mt-1 w-56 rounded-md border border-border bg-background p-2 shadow-lg"

The expanded day cell is z-10, the popup is z-20. These should be visible. BUT if the calendar grid container or any intermediate div has overflow:hidden, the popup WILL be clipped, making it appear the calendar is "cut off".

CURRENT STATE CHECKING — no overflow:hidden found in:
  - globals.css
  - layout.tsx (root or admin)
  - admin/page.tsx
  - dashboard-inner.tsx (DashboardCalendar section)

The clipping is therefore likely:
  A) The calendar grid itself (in dashboard-inner.tsx) is NOT using overflow-x-auto, causing horizontal overflow on narrow windows
  B) The calendar component is NOT using a minimum height constraint, so it appears "half height" when there are few or no tasks (just dot indicators)
  C) The expanded day popup could be clipped if there's an overflow issue between the calendar section (z-20) and the calendar container

RECOMMENDATION:
  1. Wrap the calendar grid in <div className="overflow-x-auto"> to prevent horizontal overflow
  2. Add minimum height to calendar cells: currently style={{minHeight: '48px'}} on each cell — this is fine
  3. Add className="overflow-visible" on the calendar section wrapper to ensure expanded popups are NOT clipped

=== ISSUE 2: CALENDAR TASKS NOT CLICKABLE ===

Current code in DashboardCalendar (line 211-224 of dashboard-inner.tsx):
  <Link
    href={`/admin/tasks/${task.id}`}
    onClick={(e) => e.stopPropagation()}
    className="block rounded-sm px-1.5 py-1 transition hover:bg-muted/30"
  >

ROUTE PROBLEM: The route `/admin/tasks/${task.id}` DOES NOT EXIST.
Verified: The only file under /app/admin/tasks/ is `page.tsx` (task listing page).
There is no `/admin/tasks/[taskId]/page.tsx` route.

Existing task detail route is at: `/admin/clients/[clientId]/projects/[projectId]/tasks/[taskId]/page.tsx`
This requires clientId, projectId, and taskId — all dynamic segments.

DashboardTaskSections (line 261-282 of dashboard-inner.tsx) handles this correctly:
  const taskHref = task.project_id && clientId
    ? `/admin/clients/${clientId}/projects/${task.project_id}/tasks/${task.id}`
    : undefined
  return taskHref ? <Link href={taskHref}>...</Link> : content (plain text)

But DashboardCalendar (line 211-224):
  Directly uses `/admin/tasks/${task.id}` — which 404s

RECOMMENDATION:
  Two possible approaches:
  A) Create a new simple task detail route at /admin/tasks/[taskId]/page.tsx — a quick lookup page that fetches task data from Supabase and shows a detail view. This is the simplest fix.
  B) Match the complex route pattern used in DashboardTaskSections — BUT this would require the TaskForCalendar type to include project_id and client_id, which it does NOT (it only has projects: { name: string }).
  
  Approach A is simplest and most maintainable. Create a dedicated task detail page.
